# Circle-scoped mobile bottom navigation

On mobile, Spend Circle replaces the desktop horizontal, `overflow-x-auto` Circle tab bar with a fixed bottom navigation bar plus a "More" bottom sheet (issue #124). The bar holds the three day-to-day destinations — Dashboard, Transactions, Search — and a More entry that opens a Base UI `Dialog` bottom sheet (the same shell as the filter panel) carrying the management surfaces, Categories and Members. Below the `sm` breakpoint the scrolling tab bar is gone; at `sm` and up the existing five-tab horizontal nav is unchanged.

The decision this ADR records is the one a future reader will question: **the bottom bar is Circle-scoped, not app-global.** It is rendered by `circle-layout.tsx` and exists only inside a resolved Circle route, so it appears only when there is a Circle to navigate within. Home (`/`) and Settings (`/settings`) — the Circle-selection and account surfaces — show no bottom bar. This matches the URL-as-source-of-truth model of ADR 0016/0017: Circle context is derived from the route via `useResolvedCircle`, and there is deliberately no persisted "current" or "last-viewed" Circle anywhere in the app.

The app shell still has one app-global element — the header — which on all viewports collapses its right side into an avatar dropdown (display name/email, Settings, Sign out). The bottom bar and the header are therefore split by responsibility: the header carries app-level identity and account actions everywhere; the bottom bar carries Circle-scoped destinations only where a Circle is in scope. The already-reserved `pb-24 sm:pb-6` on the protected layout's `<main>` clears the bar on mobile.

## Considered alternatives

- **App-global bar with a remembered last-viewed Circle.** A persistent bottom bar visible across the whole authenticated app, where Dashboard/Transactions/Search target the most recently opened Circle even from Home or Settings. This is the familiar consumer-app pattern, but it requires introducing a persisted "current Circle" concept that does not exist today and runs against ADR 0016/0017's URL-derived Circle resolution. It also creates ambiguous states — what the bar targets on first load before any Circle is opened — for a v1 whose primary mobile flows already start by picking a Circle on Home. Rejected as scope and state we do not need.
- **App-global bar falling back to the Personal Circle.** Same always-visible bar, but when no Circle is in scope the Circle-scoped slots target the User's Personal Circle (which always exists, so no persistence is needed). Simpler than last-viewed, but it silently jumps the User into a different Circle than the one they were last in, which is more surprising than showing no bar at all on Home/Settings. Rejected.
- **Keep the scrolling tab bar on mobile.** The status quo — no new components. Rejected because horizontal scroll hides destinations off-screen with no affordance and is the exact complaint that opened the issue.
- **Put all five destinations in the bar.** Five equal slots on a ~360px screen crush touch targets and labels. Reserving the fourth slot for More keeps the primary three legible and gives the management surfaces a roomier sheet. Rejected.

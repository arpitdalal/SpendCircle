# Circle Setup is mandatory, with completion tracked by an explicit flag

Completing Circle Setup is **required** for every regular Circle, and "setup completed" is recorded by a **dedicated `circles.setupCompletedAt` flag** rather than inferred from the presence of `setupAnswers`. The Setup step can no longer be skipped: a single **Finish** action always runs `completeCircleSetup`, which seeds the starter Categories — choosing no answers still seeds the nine shared defaults, so no Circle is ever left empty. Until a Circle is complete, every Circle-scoped route redirects to `/setup`. The Personal Circle is exempt: it has no setup concept and is born complete (`setupCompletedAt` set at bootstrap).

The shortcut being corrected: the foundation modelled "setup done" as `setupAnswers !== undefined`. That single signal is lossy — it collapses four distinct states (Finished with answers, Finished with none, never-reached, and the Personal Circle) into one, and it conflates two facts that are not the same thing: *did the owner pass the setup step* (a workflow milestone) versus *what did the owner answer* (domain data the owner can re-edit later, CS-2). That conflation is the root cause of the CS-2 bug ([PR #153](https://github.com/arpitdalal/SpendCircle/pull/153#discussion_r3431471099)): `updateCircleSettings` writing `setupAnswers` flips `undefined → defined` without running `completeCircleSetup`, and because `completeCircleSetup` treats any defined `setupAnswers` as "already complete", the one-shot starter-Category seeding is locked out for good. A separate flag dissolves the bug instead of papering over it — `setupAnswers` goes back to being pure data, and completion is its own fact.

Removing Skip is a deliberate reversal of the original "setup is skippable from anywhere" property. The reasoning is that setup pays off entirely in the starter Categories it derives; an owner who skipped left with zero Categories and could not record an expense until hand-creating one. With Skip gone, **Finish** is the only exit and it always seeds a usable set (the answers only *add* to the defaults — `residence` + leased/owned contributes Rent/Mortgage; every other purpose yields the same nine). The forced step is therefore a soft gate: it costs one click on a screen the owner sees once, and returns a ready-to-use Circle.

Enforcement is server-side, not merely routed. The `/setup` redirect is client-side UX; `updateCircleSettings` and `completeCircleSetup` are directly callable Convex mutations, so the one-shot invariant is enforced at the mutation boundary per [ADR 0015](0015-server-side-permissions-and-security-baseline.md) — a direct caller cannot bypass it through the router. PR #153 ships an interim form of this guard keyed on `setupAnswers === undefined` (the only signal available before the flag exists); this decision **re-homes that guard and `completeCircleSetup`'s "already complete" check onto `setupCompletedAt`**, which additionally allows a completed-but-answerless Circle to edit answers later without tripping the guard.

The consequences are accepted. The flag is a second field that must be set wherever a Circle becomes complete (`completeCircleSetup`, Personal-Circle bootstrap), and existing Circles must be **grandfathered** — a one-time backfill sets `setupCompletedAt` on every Circle created before this change, so the new gate applies only going forward and no current Member is trapped on `/setup`. After backfill, the field is required and nullable: `null` means an incomplete regular Circle, while a timestamp means complete. The alternative — keep inferring from `setupAnswers` now that Skip is gone — was rejected: presence-as-completion is reliable only as long as nothing else ever writes the answers, which `updateCircleSettings` does precisely by design. The full slice is specified in [`docs/issues/CS-5-enforce-circle-setup.md`](../issues/CS-5-enforce-circle-setup.md).

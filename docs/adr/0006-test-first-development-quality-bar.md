# Test-first development quality bar

Spend Circle uses test-first development with robust Vitest unit tests, integration tests, and Playwright end-to-end coverage as a project constraint. Domain helpers, Convex permission paths, histories, lifecycle rules, and critical responsive UI flows should be covered so future changes and refactors do not depend on manual regression testing.

Local and dev environments may use seed data with fake Circles, Members, Transactions, and Categories for tests, screenshots, and development. Tests must not hit production data.

Mocking has one switch and a clear boundary. Identity is handled by a dev-only auth bypass that injects an authenticated session, while MSW (Mock Service Worker) is a first-party mocking layer that intercepts outbound third-party API calls to Resend, PostHog, and Sentry; MSW does not simulate the Google OAuth redirect flow, and production authentication remains Google-only. A single `VITE_MOCKS` flag flips the running app into mock mode, enabling both the MSW worker and the auth bypass together. Default local development runs against real Google sign-in and real vendors so authentication is exercised before production; the `dev:mocks` script (which sets `VITE_MOCKS=true`) opts into mock mode, and Playwright end-to-end tests always run in mock mode so they never drive the OAuth flow. This `VITE_MOCKS` switch replaces the earlier separate `VITE_AUTH_MODE=dev` auth-mode flag.

MSW handlers live in the `packages/mocks` workspace package, a devDependency, exporting shared `handlers`, a browser `worker`, and a node `server`. The browser app registers the worker only behind `VITE_MOCKS` through a dynamic import so the bundler eliminates all mock and bypass code from the production build, while Vitest node unit and integration tests enable the `server` unconditionally in their setup. Mocks let tests assert vendor payload shape — for example Feedback email contents and the absence of financial content in analytics — without anything escaping to real vendors.

# Test-first development quality bar

Spend Circle uses test-first development with robust Vitest unit tests, integration tests, and Playwright end-to-end coverage as a project constraint. Domain helpers, Convex permission paths, histories, lifecycle rules, and critical responsive UI flows should be covered so future changes and refactors do not depend on manual regression testing.

Local and dev environments may use seed data with fake Circles, Members, Transactions, and Categories for tests, screenshots, and development. Tests must not hit production data.

Mocking has one switch and a clear boundary. Identity is handled by a dev-only auth bypass that injects an authenticated session, while MSW (Mock Service Worker) is a first-party mocking layer that intercepts outbound third-party API calls to Resend, PostHog, and Sentry; MSW does not simulate the Google OAuth redirect flow, and production authentication remains Google-only. A single `VITE_MOCKS` flag flips the running app into mock mode, enabling both the MSW worker and the auth bypass together. Default local development runs against real Google sign-in and real vendors so authentication is exercised before production; the `dev:mocks` script (which sets `VITE_MOCKS=true`) opts into mock mode, and Playwright end-to-end tests always run in mock mode so they never drive the OAuth flow. This `VITE_MOCKS` switch replaces the earlier separate `VITE_AUTH_MODE=dev` auth-mode flag.

MSW handlers live in the `packages/mocks` workspace package, a devDependency, exporting shared `handlers`, a browser `worker`, and a node `server`. The browser app registers the worker only behind `VITE_MOCKS` through a dynamic import so the bundler eliminates all mock and bypass code from the production build, while Vitest node unit and integration tests enable the `server` unconditionally in their setup. Mocks let tests assert vendor payload shape — for example Feedback email contents and the absence of financial content in analytics — without anything escaping to real vendors.

## Mock only at true boundaries, never our own logic

A test must exercise the real business logic and the real wiring between our own modules. Mock **only** at a genuine boundary: a third-party dependency, or a lower-level dependency that actually needs it (network, IO, a vendor component that cannot run in the harness). Never mock the implementation of one of our modules, and never stub the return value of a related hook/module to feed a component canned data — that fakes the exact wiring the test should prove and drifts from reality the moment the real code changes.

Concretely, for a React route test the only thing doubled is `convex/react` (the reactive client — the network boundary). The real `~/lib/data.js` hooks, the real domain logic, and the real route run against it, so a drift between the route, the data layer, and the backend query contract is actually caught. Route context (e.g. the resolved Circle from `useCircle`) is supplied through the real framework seam — a real React Router `<Outlet context>` — not by mocking our own module. The query doubles model the **backend contract** (dispatch by the function/args, filter as the server would), so the test fails if the route stops requesting the right data. This boundary lives in one shared helper, `apps/web-app/app/test/convex-react.tsx`: install it with `vi.mock("convex/react", async () => (await import("~/test/convex-react.js")).convexReactMock)` and drive each test's backend state through `configureConvex({ … })`. Reach for that helper rather than hand-rolling the doubles in a new file.

Sanctioned boundary mocks (not the smell): doubling an unrunnable vendor component while returning **real seeded data** (e.g. stubbing Better-Auth-backed `getCurrentUserOrNull` under `convex-test`, which can't run the auth component, while the user it returns is a real row in the simulated db, so handlers/db/indexes/history all run for real); and injecting only the **outbound** effect seams of a pure primitive under direct unit test (navigation, a UI toast, the error-reporting sink), where no inbound business logic is faked.

## Extract shared test scaffolding, do not redefine it per file

Tests must not redefine the same setup again and again with only slightly different data. When the same doubles, fixtures, render wiring, or contract-modeling logic would otherwise be copied across files — even just two — extract it into one well-designed lower-level helper and have every test drive it through arguments. The shared `convex-react.tsx` helper above is the worked example: the doubled network boundary, the `configureConvex` state knobs, the backend-contract filter, the Circle-context render helper, and the branded-id minter each have exactly one definition, and the per-file `setup` only supplies the data that test cares about.

This is not mechanical de-duplication. Think hard about *where* the seam belongs and *what* the configuration surface should be before extracting: the helper should encode the real contract once (so it cannot drift between files), expose a small, intention-revealing options shape (not a grab-bag of overrides), and keep domain fixtures with the suite that owns them while centralizing the plumbing. A copy-pasted `setup` that each file tweaks is a smell — the contract is then implicitly re-specified in every file and drifts the moment one copy is edited. One source of truth, configured by arguments, is the bar.

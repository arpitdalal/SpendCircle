# TanStack Form for shared web and native form state

Spend Circle standardizes on [TanStack Form](https://tanstack.com/form) (`@tanstack/react-form`) as the single form-state convention across the codebase, replacing the per-field `useState` boilerplate currently hand-rolled in each form (Transactions, Categories, and every form to come). The driver is not just removing boilerplate but establishing one form mental model — field state, dirty tracking, blur/submit validation timing, error reveal — that holds for `apps/web-app` today and the planned React Native app later. TanStack Form is headless, controlled-by-default, and renderer-agnostic, so the same `useForm` shape, the same validators, and the same domain schemas drive both platforms; only the leaf field components differ (DOM inputs on web, `TextInput` on native).

Validation reuses the existing Zod schemas (ADR 0010) directly via Standard Schema — no resolver adapter — and the server remains the authority (ADR 0015); the form is a courtesy mirror. To keep the form library fed with a shape it can actually validate per-field, the domain layer splits the transforming `transactionInputSchema` into a pure-input `transactionFormSchema` (no transform, what the form binds to and validates) plus an explicit `toMutationArgs` transform (form values → Convex mutation args, e.g. major-unit amount string → integer minor units per ADR 0009, and `paidBy` self-collapse). Both halves are framework-agnostic and live in `packages/domain`, so the future native form imports them unchanged; the web-only shadcn `Field` components stay in `apps/web-app`. The split also lets the seam be typed end-to-end, removing the lone `as` cast the single-schema approach forced.

The house validation-timing convention is on-blur-then-live: field-level `onBlur` validators give early feedback, errors are revealed only when `(isBlurred && isDirty) || submissionAttempts > 0` — so tabbing through an untouched field stays silent and "required" surfaces at submit, while an edited-but-invalid field flags on blur — and a form-level `onSubmit` validator against the full `transactionFormSchema` is the authoritative gate. Non-text fields (Category chips, Paid By select) bind through `field.handleChange` and validate on change/submit rather than blur.

Web implements the shared field kit with TanStack Form’s `createFormHookContexts` + `createFormHook` (`apps/web-app/app/lib/form.tsx`): pre-bound leaf components share one `useFieldReveal()` helper implementing `(isBlurred && isDirty) || submissionAttempts > 0`; `AmountField` only calls `handleChange` after blur when `onBlurNormalize` returns non-null so an empty untouched amount does not flip dirty on tab-through. `SelectField` keeps Paid-by-specific props optional. `formComponents` includes the transaction submit row. Exports: `useAppForm`, `useTypedAppFormContext` (TanStack’s supported pattern for typed subtrees under `AppForm`). `withForm` is not re-exported: the Transaction form’s Type (edit) + Category blocks live in dedicated modules instead; Category uses `useTypedAppFormContext` with the module-scope `transactionFormContextOptions` from `transactionFormOptions()` so `form.Subscribe` / `form.Field` stay typed to `TransactionFormValues` without prop-drilling the form handle. One `transactionFormOptions()` declaration feeds both `useAppForm` and `useTypedAppFormContext`, so the typed subtree cannot drift from the form it reads.

## Considered alternatives

- **React Hook Form** — the more mature, more popular default, and shadcn's original `Form` integration. Rejected primarily on the native goal: RHF's core advantage is *uncontrolled* DOM inputs via `register()`, which React Native cannot use, forcing a `<Controller>` around every field and diverging the web and native binding styles. This app's forms are custom-controlled-widget-heavy anyway, so RHF would pay the controlled-path cost on web while never cashing its uncontrolled edge. shadcn now ships a TanStack Form integration as well, neutralizing the remaining web-DX argument.
- **Conform** — server-action and progressive-enhancement first. Eliminated by the SPA mode of ADR 0017 (no server `action`) and irrelevant to a native client.
- **Formik / Final Form** — stagnant maintenance and no first-class Zod/Standard-Schema path.

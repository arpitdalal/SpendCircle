# End-to-end tests run against a self-hosted Convex backend

Playwright end-to-end tests run against a **real, ephemeral, self-hosted Convex backend** — the open-source `ghcr.io/get-convex/convex-backend` Docker image, booted fresh per CI job — not against hand-written fixtures and not against the production or a shared developer deployment. The thing under test is the real path: browser → real Convex queries and mutations → real database, with server-side permissions, lifecycle rules, and the history audit all exercised. This realigns the project with the PRD's own testing decisions (a real seeded non-prod backend, an OAuth-only auth bypass, vendor-only MSW) and corrects a shortcut the initial scaffold took.

The shortcut being corrected: the foundation's `VITE_MOCKS` mode both injects a fake client session *and* skips every Convex query, returning fixtures. That renders the UI against canned data and exercises zero backend code — a component-render harness wearing an end-to-end costume. It proves nothing about queries, mutations, permissions, or the frontend↔backend contract, which is the entire point of E2E. Fixtures are kept, but demoted: they serve fast component/render tests and offline UI development (`VITE_MOCKS`), and are no longer the E2E source of truth.

Exactly two things are faked in an E2E run, and the boundary is deliberate. First, **Google OAuth** — which Playwright cannot drive — is replaced by a flag-gated email-and-password sign-in that mints a *real, backend-trusted* session. Production remains Google-only (ADR 0002); the bypass is `emailAndPassword` enabled inside `createAuth` **only when `E2E_TEST_AUTH=1`**, an environment variable that is never set on the production deployment, so the path is eliminated there. A purely client-side fake session cannot work, because the backend validates the Convex JWT — the bypass must be backend-issued. Second, **outbound vendors** (Resend, PostHog, Sentry) are intercepted by MSW so no email or analytics escapes (ADR 0006, ADR 0008, ADR 0013). Everything between — Convex functions, the Better Auth component, permissions, the database — is real. This decouples what `VITE_MOCKS` had fused: E2E wants vendor-MSW **on** but the fake-session/query-skip **off** (it uses the real session path against the real backend).

The session bypass and the self-hosted backend were both spiked and proven before this decision (see the throwaway `spike/selfhost-convex/`). The backend image boots, accepts `convex deploy` including the Better Auth component, and serves live auth routes (`GET /api/auth/ok` → `{"ok":true}`). The bypass round-trip is green end to end: `POST /api/auth/sign-up/email` mints a session (the `onCreateUser` trigger creates the User + Personal Circle), `GET /api/auth/convex/token` exchanges it for an EdDSA Convex JWT whose issuer matches `auth.config.ts`, and an authenticated `listMyCircles` then returns exactly that user's Personal Circle — confirming identity flows through `getAuthUser` → `requireCurrentUser`.

Self-hosting in CI was chosen over the alternatives because each PR job gets its own fresh container: isolated, ephemeral, parallel-safe, deterministic, free, and structurally incapable of touching production data (ADR 0007). Keeping fixtures as the E2E surface was rejected — it tests nothing across the seam. A dedicated cloud dev deployment is a viable fallback but carries shared-state flakiness, is not free to parallelize, and needs seed/reset coordination across jobs. Convex preview deployments are possible but reverse the "defer per-PR previews" stance (ADR 0007) and re-introduce cloud coupling for every test run. The deferred-previews decision concerned *deploy* previews, not the test path, so it is not contradicted here. The one operational obligation is pinning the backend image tag to a build compatible with the `convex` npm version (`^1.26.2`); a mismatch is the only failure mode the spike surfaced, and it fails loudly at `convex deploy`.

The consequences are accepted. E2E is slower than rendering fixtures and requires maintaining the gated bypass and CI orchestration (boot, admin key, deploy, build, serve, run). In exchange, a green E2E run means the real contract works. Per-feature seed data is built with the features that need it rather than up front (the same deletion-test discipline applied across the implementation slices); a fresh E2E user already has a Personal Circle from `onCreateUser`, which is sufficient for the current app-shell and deep-link flows. As feature slices land, their E2E coverage seeds the data each one needs.

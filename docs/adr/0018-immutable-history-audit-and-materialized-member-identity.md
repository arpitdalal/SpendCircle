# Immutable history audit and per-Circle materialized member identity

Spend Circle records every Circle, Transaction, and Category change as an append-only, immutable audit: one history row per user action — the event is the row — holding the audited entity's id (`entityId`), the acting Member (`actorMemberId`, optional for system actions), an `action` verb, a `createdAt`, and `changes` as an array of `{ field, from?, to? }` whose values are human-readable strings formatted once at write time — money via the Circle Currency (ADR 0009), dates as plain `YYYY-MM-DD` with no timezone conversion, Members as their Display Name, Categories as their names (`from` is absent on a *created* event, `to` on an *archived* one). Stored values are never re-resolved, so a history line always shows what was true when it was written; a single entity's history is read by its id newest-first, and grouping is intrinsic because each event is one row. Histories are written server-side only (ADR 0015) and never store or render raw internal IDs inside `changes`. We considered a structured, reference-based history that stored entity ids in each change and re-resolved display values — and live identity — at read time, and rejected it: it pushed identity rendering and a "no raw ID leaks" invariant into every read path, coupled an immutable audit to mutable live data, and bought nothing an audit log should want. Immutability is the point.

Because history text is frozen, live identity lives on each entity's *current* display instead of in the audit. Each `members` row is the per-Circle materialized identity for a User: it carries `displayName`/`image` (not "snapshot" columns) that mirror the User's current Google profile (ADR 0002) while the membership is active, freeze when the Member is removed, and refresh on rejoin. There is exactly one member row per (Circle, User) — leaving flips its status to `removed`, rejoining reactivates the same row — so member ids are stable and no duplicate rows are ever created. A single propagation path updates a User's active member rows when their profile changes and leaves removed rows untouched. Transaction Paid By / Recorded By and the Member List read this materialized identity — current while active, the frozen value while removed, current again on rejoin — which is what satisfies the Removed Member and rejoin requirements (CONTEXT: Removed Member, Paid By; PRD stories 42–44). The immutable history, by contrast, keeps the name as it read at the moment of the event.

This decision is implemented. The `members` table carries `displayName`/`image` (the materialized identity), and `histories` is the event-as-row shape `{ entityId, actorMemberId?, action, changes: { field, from?, to? }[], createdAt }` indexed `by_entity`. The history audit is written only through `recordEvent` in `packages/convex/convex/history.ts` (with typed `*Entity` constructors), and the single propagation path is `propagateUserProfile` in `model.ts`, invoked by the `onUpdateUser` Better Auth trigger in `auth.ts`; both are covered by `history.test.ts` and the `propagateUserProfile` suite in `users.test.ts`.

# React Router Framework Mode in SPA mode for the web app

Spend Circle's `apps/web-app` uses React Router Framework Mode in SPA mode (`ssr: false`) as a first-class browser/data router with real, declared paths, not a memory router or hand-rolled routing bolted on later. Framework Mode is chosen for type-safe route params, generated `href()` typed links so paths are never hand-written, automatic per-route code splitting via lazy route modules, and nested layout routes for the app shell and auth boundary. This supersedes the plain "Vite, React, TypeScript" framing of ADR 0004: the `@react-router/dev` plugin owns the Vite build, the HTML entry, and the root route, so Framework Mode is the app's build and routing model rather than an add-on. Routes are config-based in `routes.ts` rather than file-based, because file-name conventions fight the `slug-id` refs and Circle-scoped nesting of ADR 0016.

SPA mode means there is no server runtime. Server `loader` and `action` do not exist under `ssr: false`, and only client loaders are available. This is consistent with ADR 0007, which serves the app as static assets on Cloudflare Workers: the build emits a static client bundle with a prerendered `index.html`, and the Worker must serve that `index.html` as the SPA fallback for unmatched paths so deep links like `/circles/my-home-c1/transactions/rent-t1` resolve client-side. We deliberately do not use loaders or actions for data, and do not use client loaders for route guards either. Convex reactive `useQuery`/`useMutation` is the single data layer (ADR 0001) and reactive auth flows through `ConvexBetterAuthProvider` (ADR 0002). The permission check for a route *is* its reactive Convex query result, so the live-update and live-revocation requirements stay satisfied with one reactive path and guard logic cannot drift into a second non-reactive copy.

The route tree splits on authentication. A public layout hosts `/signin` (the sign-in wrap of ADR 0014), `/terms`, `/privacy`, and the opaque `/invite/:token` landing (the ADR 0016 token-only exception). A protected layout gates on reactive Convex auth across three states: unauthenticated redirects to `/signin`; authenticated-but-not-yet-bootstrapped (a Google session exists but the Spend Circle User and Personal Circle do not) renders its own gated onboarding branch; authenticated-and-ready renders the app shell. While auth resolves, the protected layout shows a full-page splash rather than flashing the sign-in screen.

ADR 0016 canonical-URL behavior is implemented as nested guard layout routes, resolved in stages by route depth. The Circle guard at `/circles/:circleRef` resolves, canonicalizes, and guards the Circle, falling back to the User's default safe route, and provides the resolved Circle to children via Outlet context. Object routes such as `/circles/:circleRef/transactions/:transactionRef` and `/circles/:circleRef/categories/:categoryRef` read the resolved Circle from context and resolve their own ref within it, falling back to the Circle route. Because resolution is staged, an inaccessible Circle means the object guard never mounts, so object existence cannot leak. All guards share one primitive — parse the trailing-segment ID, subscribe to the object by ID, then either show pending UI, canonicalize a stale slug with replace navigation, or fire the generic unavailable-link snackbar and navigate to the fallback — parameterized only by fallback target. Ref parsing and canonical-path construction stay in `packages/domain` per ADR 0016. A root catch-all splat route follows the same generic, non-revealing behavior and lands the User on their default safe route with the unavailable-link snackbar; there is no dedicated 404 page.

Implementation refinement: the one shared guard primitive is realized as `useResolvedRef`, the resolution *state machine* — it interprets the parsed-ref result plus the fetched value into pending/ready, canonicalizes a stale slug with replace navigation, and on a gone target fires the generic unavailable-link snackbar and navigates to the fallback. Per-object adapters (`useResolvedCircle`, and the future `useResolvedTransaction`/`useResolvedCategory`) own reading the route param, parsing the ref, subscribing through Convex `useQuery`, and mock synthesis, then feed the primitive a `parsed` flag and the resolved value; moving the subscribe into the adapter keeps the primitive free of Convex and the mock flag, so its interface is its test surface. A gone target splits internally into two classes that stay identical to the User — preserving the anti-enumeration parity of ADR 0016 — but differ in telemetry: an inaccessible-or-missing object is an expected permissions/visibility outcome and is silent, while an unparseable ref (its trailing segment did not parse — almost always a bad link the app itself emitted) is additionally reported as a warning-level application error (Sentry once it lands, ADR 0012) so it surfaces as a bug to fix rather than vanishing into the same silent path.

Implementation refinement: the unavailable-snackbar copy is a **closed vocabulary, not free text**. `snackbar.tsx` owns a `UNAVAILABLE_MESSAGES` record (`link`, `circle`, …) and exposes `showUnavailable(target)`; an adapter passes a *token*, never a string (`useResolvedCircle` passes `"circle"`; the splat route and the future object guards default to `"link"`). This keeps ADR 0016 enforcement structural rather than aspirational: every non-revealing message lives in one greppable place that cannot drift or leak per call site, a typo is a compile error rather than a leak, and within a token the same copy still covers both missing and inaccessible — so the anti-enumeration parity above holds even though the copy is now object-flavored (a Circle that "isn't available" reads correctly for a Member removed mid-session, who never clicked a "link"). Arbitrary snackbar copy outside this family still goes through the unconstrained `show(message)`; the constraint is deliberately scoped to the anti-enumeration path. This is also why an *archived* (still-accessible) Circle does **not** flow through this path at all — the Member keeps access, so its routes surface read-only in place (the existing archived banner, and forms gated on `writable` so an open create form closes live when the Circle is archived mid-edit) rather than ejecting to a fallback.

Transaction route state is URL-owned when reload persistence matters. The Monthly Ledger's selected month is the `month=YYYY-MM` query parameter; malformed or unsupported UI query state is normalized with replace navigation rather than treated as an unavailable object. The Transactions route canonicalizes missing default state to `month=<current>&status=all&type=all`, where `status` is the Ledger Filter lifecycle scope (`active`, `archived`, or `all`) and `type` is the Transaction type filter (`all`, `expense`, or `income`). Ledger Filter-owned params are colocated with the Ledger Filter code and removed when the selected month changes, so the month switch preserves route-owned state without carrying stale filters across month boundaries. Ledger Filter edits are draft state in a modal filter panel and write URL/query state only when the User applies them; Reset immediately writes canonical defaults. The Add Expense/Add Income forms are non-object UI state, so they deep link as `/circles/:circleRef/transactions?month=YYYY-MM&new=expense|income`; closing or saving removes `new` and preserves `month`, and reload opens a fresh form rather than restoring unsaved draft fields. Edit forms use the canonical object route `/circles/:circleRef/transactions/:transactionRef/edit?month=YYYY-MM`; the Transaction ref is `buildRef(transaction.title, transaction.id)`, stale title slugs canonicalize with replace, and the edit target is fetched by ID rather than found in the visible ledger page. The `month` query remains ledger context even if the edited Transaction belongs to another month, so closing or saving returns to `/transactions?month=...` instead of auto-jumping to the Transaction's own month; Ledger Filter params are not propagated through Transaction detail or edit routes.

Transaction Search is a separate Circle route at `/circles/:circleRef/search`, not part of the Transactions route. The Search route canonicalizes missing default state to `status=all&type=all`, shows paginated Circle Transactions (active and archived, with archived rows visually distinguished) immediately, and treats an absent date range as all-time. No-text results are date-desc through the Transaction indexes; non-empty text results are paginated by Convex full-text relevance via the Transaction search projection (synced transactionally on create, edit, archive, and restore). Only non-default narrowing filters such as text, Categories, Recorded By, Paid By, exact date range, and amount range add query parameters beyond the canonical defaults. Search edits are draft state and write URL/query state only when the User submits Search; Reset immediately writes canonical defaults. Search rows may link to Transaction detail through the existing Circle-scoped object route, but detail Back returns to Transactions until a future explicit return-path mechanism exists.

An edit deep link means "open an editable active Transaction," not Transaction detail or archive context. Missing, inaccessible, wrong-Circle, archived, or not-editable Transactions all resolve through the same unavailable-link snackbar and fallback to the Circle's Transactions route with the selected month preserved; Owners do not gain field-edit access through an edit URL. Archived Circles are still accessible and stay on the read-only route surface rather than going through unavailable-link fallback.

# Spend Circle V1 PRD

## Problem Statement

People often track money with partners, families, roommates, friends, or trip groups, but most lightweight finance tools are either private-only, spreadsheet-like, or optimized for settlement instead of shared budget visibility. Users need a trustworthy live app where multiple people can record and review Transactions in shared Circles without losing history, rewriting each other's records, or manually refreshing data.

## Solution

Build Spend Circle v1 as a responsive web app for collaborative Transaction tracking. Users sign in with Google, get an always-solo Personal Circle, and can create regular Circles for shared tracking. Members can record Expense and Income Transactions, classify them with one or more type-specific Categories, search and export data, view dashboards and Monthly Ledgers, and receive focused in-app Notifications. Convex powers live updates and server-side permission enforcement.

## Implementation Decisions

- Use the glossary in `CONTEXT.md` as canonical vocabulary: Circle, Personal Circle, Member, Owner, Transaction, Recorded By, Paid By, Category, Monthly Ledger, Ledger Filter, Dashboard, Transaction Search, Export, Notification Center, Feedback.
- Use a pnpm workspace monorepo.
- Use `apps/web-app` for the authenticated responsive web app.
- Reserve future `apps/site` for marketing and future `apps/native-app` for React Native.
- Use `packages/convex` for Convex schema, functions, generated API, and backend-side business behavior.
- Use `packages/domain` for pure shared primitives: supported currencies, money helpers, plain date/month helpers, color palettes, validation constants, and Zod schemas.
- Keep `packages/domain` free of React, Convex, browser-only, and platform-specific imports.
- Use `packages/mocks` (devDependency) for first-party MSW mocking: shared vendor `handlers`, a browser `worker`, and a node `server`, reusable across the web app runtime, Vitest setup, and a future native client.
- Use Vite, React, TypeScript, Tailwind, shadcn/ui, and Recharts.
- Use React Router Framework Mode in SPA mode (`ssr: false`) for app routing, as a first-class browser/data router with real declared paths in config-based `routes.ts` (not a memory router or hand-rolled routing, not file-based routing); see ADR 0017.
- Treat React Router as the web app's build and routing model: the `@react-router/dev` plugin wraps Vite, owns the HTML entry and root route, and provides type-safe route params, generated `href()` typed links (no hand-written path strings), automatic per-route code splitting, and nested layout routes.
- Run with no server runtime under SPA mode; server loaders/actions do not exist; deploy as a static client build with `index.html` served as the SPA fallback for unmatched paths (ADR 0007).
- Do not use React Router loaders/actions for data, and do not use client loaders for route guards; Convex reactive `useQuery`/`useMutation` is the single data layer and a route's permission check is its reactive query result, keeping live updates and live revocation reactive with no second non-reactive guard path.
- Split routing on authentication: a public layout (`/signin`, `/terms`, `/privacy`, opaque `/invite/:token`) and a protected layout that gates on reactive Convex auth across three states — unauthenticated (redirect to `/signin`), authenticated-but-not-bootstrapped (own gated onboarding branch), and authenticated-and-ready (app shell) — showing a splash while auth resolves.
- Implement ADR 0016 canonical URLs as nested guard layout routes resolved in stages by route depth: the Circle guard at `/circles/:circleRef` resolves, canonicalizes, and guards the Circle (fallback to default safe route) and provides it via Outlet context; object routes resolve their own ref within the resolved Circle (fallback to the Circle route). One shared primitive (parse trailing-segment ID, subscribe by ID, then pending UI, stale-slug replace navigation, or generic unavailable-link snackbar plus fallback) is parameterized only by fallback target; ref parsing and canonical-path construction stay in `packages/domain`.
- Use a root catch-all splat route with the same generic, non-revealing fallback to the default safe route plus unavailable-link snackbar; no dedicated 404 page.
- Use Notion-style `slug-id` app URLs where the ID is authoritative and stale or missing slugs canonicalize with router replace navigation after permission checks.
- Keep Circle object routes Circle-scoped, such as `/circles/:circleRef/transactions/:transactionRef` and `/circles/:circleRef/categories/:categoryRef`; do not support global object detail routes.
- Show a generic unavailable-link snackbar and route to a safe fallback when deep links are missing or inaccessible without revealing which case occurred.
- Use Node LTS tooling, pnpm, TypeScript strictness, and Biome for lint/format.
- Do not use Next.js, Effect, Bun, or Deno in v1.
- Use Convex for live data, database, realtime subscriptions, mutations, and server-side permission checks.
- Use Convex functions as the shared backend contract for future web and native clients.
- Do not build a custom REST API or custom WebSocket server in v1.
- Add HTTP endpoints only for integrations, webhooks, or public API needs.
- Deploy the web app to Cloudflare Workers with static assets.
- Deploy backend code to Convex Cloud.
- Use personal Convex dev deployments locally and production Convex deployment from GitHub Actions on main.
- Skip automatic per-PR Convex preview deployments initially.
- Use Better Auth with Google-only sign-in.
- Tie User identity to Google provider subject, not email.
- Use Google Account Email for Invitation matching.
- Use sign-in-wrap copy on the Google sign-in screen for Terms and Privacy Policy acceptance.
- Store accepted terms version, privacy version, and accepted timestamp on first sign-in.
- Host `/terms` and `/privacy` in `apps/web-app` initially; move to `apps/site` later if needed.
- Create a Personal Circle automatically for every Google-authenticated User.
- Keep Personal Circle always solo: exactly one Member, no invites, no archive/delete/leave/transfer.
- Allow regular Circles to start solo and later invite Members.
- Allow duplicate Circle names; use Circle Mark and Circle Color for distinction.
- Allow empty regular Circle deletion only when it has exactly one Member and no Transactions ever created.
- Archive regular Circles once any Transaction exists instead of deleting.
- Revoke pending Invitations and invalidate Invitation Links when a Circle is archived or deleted.
- Enforce Circle Visibility server-side.
- Use Circle Setup to derive starter Categories and confirm Currency.
- Use a checked-in supported ISO 4217 currency list; default from locale with USD fallback.
- Validate Currency server-side and lock it after the first Transaction exists.
- Store Amounts as positive integer minor units; UI supports two decimals, no zero/negative, max 999,999,999.99.
- Use plain date strings for Transaction Date; no timezone conversion for Dashboard/Transaction Search/month buckets.
- Store operational timestamps separately for Audit Metadata and histories.
- Add Transactions with Add Expense/Add Income CTAs plus segmented type switch.
- Require confirmation when changing a saved Transaction type.
- Clear Categories on Transaction Type Change and require active Categories for the new type.
- Use Title required and Note optional.
- Paid By defaults to Recorded By and can be any current Member.
- Field edit rights follow Recorded By only.
- Owner can archive/restore any Transaction but cannot edit another Member's Transaction fields.
- Archived Transactions are frozen, excluded from default Ledger Filter, Transaction Search, and Dashboard totals, and restorable by creator or Owner.
- Refunds are Income categorized as Refund; no linking to original Expense in v1.
- Categories are Circle-scoped, type-specific, color-required, and creator-owned for field edits.
- Owner can archive/restore any Category but cannot rename/recolor another Member's Category.
- Category names are unique per Circle and Transaction type, case-insensitive, including archived names.
- Transactions must have at least one Category and cannot contain duplicate Categories.
- Archived Categories stay attached to existing Transactions and may remain through edits, but cannot be newly added.
- Dashboard is per Circle and can filter totals by Paid By.
- Monthly Ledger is the operational Transaction management surface.
- Ledger Filter is integrated with Transactions; Transaction Search is a separate Circle nav item.
- Export is CSV per Circle, available to current Members for visible Transactions.
- Notification Center is in-app only and user-specific.
- Email Notifications are limited to Invitation and Welcome emails.
- Feedback sends email to a configured support address through Resend.
- Use Resend for transactional email.
- Use Sentry from beta with strict masked error-triggered Session Replay; no normal replay sampling.
- Keep Sentry operational monitoring enabled regardless of product analytics opt-out.
- Use PostHog for product analytics, no PostHog Session Replay in v1.
- Enable product analytics by default after signup legal acceptance; allow opt-out in Settings -> Privacy.
- Do not send Transaction Title, Note, exact Amount, Feedback free text, or other financial content to analytics.
- Enforce all auth, membership, ownership, and action permissions inside Convex functions.
- Write histories server-side only.
- Store secrets only in platform environment variables.
- Rate limits: same Invitation resend capped at 3/day per Circle/email, total invitation emails capped at 100/User/day, Feedback capped at 20/User/day, Welcome email once/User, failed Invitation Link attempts throttled without user-enumeration leaks.

## Testing Decisions

- Use test-first development as a project constraint.
- Use Vitest for unit tests.
- Use Playwright for end-to-end tests across desktop and mobile viewports.
- Use MSW as a first-party mocking layer for outbound third-party API calls (Resend, PostHog, Sentry); MSW does not simulate the Google OAuth redirect flow.
- Handle identity in mock mode with a dev-only auth bypass that injects a session; production auth remains Google-only.
- Gate mock mode behind a single `VITE_MOCKS` flag that enables both the MSW worker and the auth bypass together, exposed via a `dev:mocks` script; this replaces the earlier `VITE_AUTH_MODE=dev` flag. Default local dev uses real Google sign-in and real vendors so auth is exercised before production.
- Run Playwright end-to-end tests always in mock mode so they never drive the OAuth flow.
- Keep MSW handlers in a `packages/mocks` workspace devDependency exporting shared `handlers`, a browser `worker`, and a node `server`; register the browser worker only behind `VITE_MOCKS` via dynamic import so the production build eliminates all mock and bypass code, and enable the node `server` unconditionally in Vitest setup.
- Tests must not hit production data.
- Local/dev seed data may include fake Circles, Members, Transactions, Categories, and histories.
- Unit test pure `packages/domain` helpers: money parsing/formatting, integer minor-unit conversion, date/month helpers, currency list, color selection, validation constants, Zod schemas.
- Integration test Convex schema, queries, mutations, permissions, live-update-relevant behavior, lifecycle transitions, rate limits, and history writing.
- Test every permission path: Owner, Member, Removed Member, Recorded By, Paid By, Personal Circle owner, archived Circle, archived Category, archived Transaction.
- Test Invitation flows: create, resend, revoke, expiry, wrong Google Account Email, used link, archive/delete invalidation, rejoin.
- Test Transaction flows: create Expense/Income, edit own, reject edits by others, Owner archive/restore, type change, Archived Transaction frozen, refund as Income.
- Test Category flows: create, inline create, uniqueness, archive/restore, archived-name collision, edit rights, removed creator, multi-category behavior.
- Test Dashboard math with active/archived Transactions, archived Categories, Paid By filter, comparison ranges, and non-additive Category analytics.
- Test Monthly Ledger month/year navigation, sorting, totals, and Ledger Filter defaults.
- Test Transaction Search filters including Removed Members and Archived Categories.
- Test Export scope and optional archived inclusion.
- Test Notification creation, read state, links, inaccessible object fallback, and archived object context.
- Test Feedback rate limits and email payload shape without leaking financial content.
- Test routing behavior: auth route gating across the three states (unauthenticated, authenticated-not-bootstrapped onboarding, ready), staged Circle-then-object ref resolution, stale-slug canonicalization via replace navigation, missing/inaccessible deep-link fallback for both Circle and object without distinguishing the cases, and the catch-all splat fallback.
- E2E critical flows: first sign-in/onboarding, Personal Circle use, create regular Circle, invite stub, accept Invitation, add/edit/archive/restore Transaction, create/archive/restore Category, Dashboard drilldown, Search, Export, Notification Center, Feedback, Settings analytics opt-out.
- Regression tests are required for every fixed bug.
- Prefer testing external behavior and domain invariants over implementation details.

## Out of Scope

- Account Deletion.
- Mixed currencies inside one Circle.
- Currency conversion.
- Password/email auth.
- Apple/social providers beyond Google.
- File uploads, receipts, screenshots, or attachments.
- Settlement, splits, debts, or who-owes-whom balances.
- Budgets, limits, or envelopes.
- Recurring Transactions.
- Offline writes.
- Importing Transactions.
- Public Circle discovery.
- Marketing site implementation.
- Native mobile app implementation.
- Custom REST API.
- Custom WebSocket server.
- Per-PR Convex preview deployments.
- PostHog Session Replay.
- User-disablable operational error monitoring.
- Exporting histories.
- Uploaded Circle images.
- Editable Profile Picture in v1.
- Category icons.
